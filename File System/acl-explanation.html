Access Control List (ACL) - Educational Guide

🔐 What is Access Control List (ACL)?

An Access Control List (ACL) is a security mechanism used in file systems to control which users or groups can access specific files or directories and what operations they can perform on them.

ACL provides fine-grained permission management by maintaining explicit lists of permissions for each resource (file or directory). Each entry in an ACL specifies:

  • User/Group: Who is requesting access
  • Resource: Which file or directory is being accessed
  • Permissions: What operations are allowed (Read, Write, Execute)
💡 Key Concept: Instead of just having "owner," "group," and "others" with fixed permissions, ACL allows you to define custom permissions for any number of users or groups for each file independently.

📋 Traditional File Permissions vs. ACL

Traditional Permission Model (Unix/Linux)

Traditional permissions use a simple 3-tier model:

  • Owner (User): The person who owns the file
  • Group: Members of a specific group
  • Others: Everyone else

Each tier gets three permissions (Read, Write, Execute), represented as 'rwx' (e.g., 755 = rwxr-xr-x)

Example: -rwxr-xr-x ├─ Owner: rwx (7) ├─ Group: r-x (5) └─ Others: r-x (5)

Limitations of Traditional Permissions:

  • Can't grant Read access to user A and Write access to user B on the same file
  • Limited to owner, group, and others categories
  • If a user is in a group, they inherit all group permissions
  • Can't apply different permissions to specific users

ACL Model

ACL allows multiple explicit entries per resource, each with custom permissions.

File: report.txt ───────────────── user: alice → r-- user: bob → rw- user: charlie → r-x user: guest → --- group: admin → rwx

Comparison Table:

Aspect Traditional Permissions ACL
Granularity 3 levels (owner, group, others) Per-user or per-group entries
Flexibility Limited Very flexible
Default Permissions Same for all users in group Custom per user
Complexity Simple to understand More complex
Scalability Poor (limited levels) Excellent (unlimited entries)
Inheritance Automatic Can be configured

✨ Benefits of ACL in Modern File Systems

1. Fine-Grained Access Control

Grant different permissions to different users without affecting others. For example, allow Alice to read a file, Bob to read and write, and Charlie to execute it.

Example: Project document
User Permissions Purpose
alice (Owner) rwx Full control
bob (Editor) rw- Can edit
charlie (Reviewer) r-- Can only read

2. Easier Multi-User Collaboration

In organizations with many users, ACL simplifies permission management by allowing different permission sets for each user without creating complex group hierarchies.

3. Better Security

  • Principle of least privilege: Users get only the permissions they need
  • Deny access to unauthorized users explicitly
  • Audit trail can track who accessed what

4. Inheritance and Default Permissions

Files created inside a directory can inherit parent directory ACL entries by default, ensuring consistent security policies.

5. Support for Groups and Roles

ACL can handle both individual users and groups, allowing role-based access control.

💡 Example: Instead of listing 50 individual users, you can have one ACL entry for "qa_team" group with appropriate permissions.

6. Dynamic Permission Changes

Permissions can be updated in real-time without affecting other users or requiring system restart.

7. Cross-Platform Compatibility

Modern file systems (NTFS, ext4 with ACL extensions, etc.) support ACLs, making it easier to migrate between systems while maintaining security policies.

🔑 Permission Types

Three Basic Permission Types:

1. Read (r) - Bit 4

  • For Files: Permission to view the file's contents
  • For Directories: Permission to list directory contents
  • Required before any other operations

2. Write (w) - Bit 2

  • For Files: Permission to modify or delete the file
  • For Directories: Permission to create, delete, or modify files within it
  • Implies ability to change file properties

3. Execute (x) - Bit 1

  • For Files: Permission to run the file as a program/script
  • For Directories: Permission to enter/traverse the directory
  • For directories, execute is necessary to access files within it
⚠️ Important: For directories, 'execute' permission is needed to traverse into it. If a directory lacks 'x' permission, users cannot access files within it even if they have 'r' permission.

Permission Representation:

rwx = 111 (binary) = 7 (decimal) = Full permissions rw- = 110 (binary) = 6 (decimal) = Read + Write r-x = 101 (binary) = 5 (decimal) = Read + Execute r-- = 100 (binary) = 4 (decimal) = Read only -w- = 010 (binary) = 2 (decimal) = Write only --x = 001 (binary) = 1 (decimal) = Execute only --- = 000 (binary) = 0 (decimal) = No permissions

⚙️ ACL Implementation in This System

Features Implemented:

  • User Management: Create and manage users
  • File Management: Create files with specified owners
  • Permission Assignment: Grant rwx permissions to users per file
  • Access Control: Verify permissions before operations
  • Inheritance: Owner gets full rwx by default
  • Audit Logging: Track all operations
  • Permission Testing: Test if a user can perform an action

ACL Entry Structure:

{ userId: "user-alice", resourceId: "file-report.txt", resourceType: "file", permissions: "rwx", createdAt: "2024-01-15T10:30:00Z" }

Access Check Algorithm:

  1. Check if the user is the file owner
    • If yes → Grant full access (rwx)
  2. Search for ACL entry matching (user, file)
    • If not found → Deny access
  3. Extract permission bits for requested action
    • If bit is set → Grant access
    • If bit is not set → Deny access

Ownership and Default Permissions:

  • File creator becomes the owner
  • Owner automatically receives 'rwx' permissions
  • Only owner or admin can modify ACL entries
  • Cannot remove permissions from the owner

🌐 Real-World Examples

Example 1: Project Collaboration

Scenario: Development team working on a project
File: project-code.py Owner: alice (Lead Developer) ACL Entries: ───────────────────────── alice → rwx (Full control) bob → rw- (Can read & edit) charlie → r-- (Can read only) guest → --- (No access)

Example 2: Directory-Based Sharing

Scenario: Department document management
Directory: /finance/reports/ ACL Entries: ───────────────────────── finance_dept → rwx (All team members) external_audit → r-- (Read-only access) boss → rwx (Full access) intern → --- (No access)

Example 3: Security-Sensitive Files

Scenario: Configuration files needing protection
File: database-config.env Owner: sysadmin ACL Entries: ───────────────────────── sysadmin → rwx (Owner) app_service → r-- (Read credentials) developers → --- (No access) auditors → r-- (Read for compliance)

🎯 Best Practices for ACL Management

1. Principle of Least Privilege

Grant users only the minimum permissions they need to do their job. If someone only needs to read a file, don't grant write or execute permissions.

2. Use Groups Effectively

Instead of adding permissions for each individual user, create groups and assign permissions to groups when possible.

3. Regular Audits

Review ACL entries periodically to ensure they're still appropriate, especially for former employees or changed roles.

4. Document Permissions

Maintain documentation about why certain permissions were granted to specific users.

5. Permission Inheritance

Use directory-level ACLs to establish consistent permission policies for new files created within.

6. Avoid Excessive Permissions

Be cautious with execute permissions; they should be granted deliberately, not by default.

7. Protect ACL-Modified Objects

Regular backups should include ACL information to prevent accidental loss of permission configuration.
